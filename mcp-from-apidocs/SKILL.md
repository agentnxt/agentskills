# MCP from API Docs

Generates a **complete, ready-to-run MCP server project** from any API documentation input.

## Inputs accepted
- A URL to API documentation (will be fetched)
- Pasted API doc text (markdown, HTML, plain text)
- Uploaded file (PDF or HTML)
- OpenAPI / Swagger spec (JSON or YAML)

## Output
A complete project folder containing:
- `server.py` — the MCP server (Python, using `mcp` SDK)
- `Dockerfile` — containerised deployment
- `docker-compose.yml` — one-command local run
- `requirements.txt` — pinned dependencies
- `README.md` — setup, config, and deployment guide
- `test_server.py` — basic smoke tests for every tool
- `.github/workflows/ci.yml` — GitHub Actions: run tests + push image to Docker Hub on every push to `main`
- `.dockerignore` — keeps the image lean

## Workflow

### Step 1 — Ingest the API documentation

Depending on input type:

| Input | Action |
|---|---|
| URL | Use `web_fetch` to retrieve the page. If it's a Swagger/OpenAPI JSON/YAML URL, parse it directly. |
| Pasted text | Use as-is |
| Uploaded file | Read from `/mnt/user-data/uploads/` |
| OpenAPI spec | Parse spec for paths, methods, parameters, and schemas |

After ingesting, extract and summarise:
- **Base URL** of the API
- **All endpoints** — method, path, description, parameters, request body, response shape
- **Authentication scheme** (for this skill, always API key; note the header/query param name from docs)
- Any **pagination** patterns

If anything critical is ambiguous (e.g. the API key header name isn't clear), ask the user before proceeding.

---

### Step 2 — Design the MCP tools

Map each meaningful API endpoint to one MCP tool. Follow these rules:

- **One tool per logical action** (not per HTTP method). E.g. `search_products`, `get_order`, `create_invoice`.
- **Tool names**: `snake_case`, descriptive verb-noun pairs.
- **Tool descriptions**: Write clear, Claude-friendly descriptions — Claude will use these to decide when to invoke the tool. Be specific about what the tool does and what parameters it takes.
- **Parameters**: Use the exact types from the API docs. Mark required vs optional clearly.
- Group related read endpoints together if the API is large (>20 endpoints) — don't create more than ~15 tools unless the user asks for full coverage.
- Skip purely administrative or webhook-registration endpoints unless the user specifically asks.

Present the proposed tool list to the user as a simple table before writing code:

```
| Tool name         | Method | Endpoint         | Description                  |
|-------------------|--------|------------------|------------------------------|
| search_products   | GET    | /v1/products     | Search the product catalogue |
| get_product       | GET    | /v1/products/:id | Fetch a single product       |
| create_order      | POST   | /v1/orders       | Place a new order            |
```

Ask: "Does this look right? Anything to add, remove, or rename?" Wait for confirmation before writing the full project.

#### `server.py`

Use the official `mcp` Python SDK. Follow this structure:

```python
#!/usr/bin/env python3
"""
<API Name> MCP Server
Generated by mcp-from-apidocs skill
"""
import os
import httpx
from mcp.server.fastmcp import FastMCP

API_KEY = os.environ["<API_NAME>_API_KEY"]
BASE_URL = "<base_url>"
HEADERS = {"<auth_header>": API_KEY}  # adjust header name from docs

mcp = FastMCP("<api-name>-mcp")

@mcp.tool()
def tool_name(param1: str, param2: int | None = None) -> dict:
    """
    Clear description Claude will use to invoke this tool.

    Args:
        param1: Description of param1
        param2: Optional description of param2
    """
    resp = httpx.get(f"{BASE_URL}/endpoint", headers=HEADERS, params={...})
    resp.raise_for_status()
    return resp.json()

if __name__ == "__main__":
    mcp.run()
```

Rules:
- Always use `httpx` (not `requests`) for HTTP calls — it's already in the base image.
- Always call `resp.raise_for_status()` before returning.
- Return raw `resp.json()` unless the response is very large — in that case, extract and return only the relevant fields.
- Use `os.environ` for the API key — never hardcode it.
- For POST/PUT bodies, pass a `pydantic` model as the parameter type if the body is complex.
- Add `timeout=30` to all httpx calls.

#### `.github/workflows/ci.yml`

Creates a two-job GitHub Actions pipeline:
- **test**: runs `pytest` on every push and pull request
- **docker**: on push to `main` only — builds the image and pushes it to Docker Hub as `<dockerhub-username>/<api-name>-mcp:latest` and `:<git-sha>`

```yaml
name: CI

on:
  push:
    branches: ["main"]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
      - run: pip install -r requirements.txt pytest
      - run: pytest test_server.py -v

  docker:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: |
            ${{ secrets.DOCKERHUB_USERNAME }}/<api-name>-mcp:latest
            ${{ secrets.DOCKERHUB_USERNAME }}/<api-name>-mcp:${{ github.sha }}
```

Replace `<api-name>` with the actual API slug. Tell the user they need to add two GitHub repository secrets:
- `DOCKERHUB_USERNAME` — their Docker Hub username
- `DOCKERHUB_TOKEN` — a Docker Hub access token (create at hub.docker.com → Account Settings → Security)

#### `test_server.py`

Write one smoke test per tool using `pytest` + `unittest.mock`. Mock `httpx` calls — do not make real network requests. Each test should:
1. Mock the httpx response with a realistic fixture response
2. Call the tool function directly
3. Assert the return value has the expected shape/keys

```python
import pytest
from unittest.mock import patch, MagicMock
import server

@patch("server.httpx.get")
def test_tool_name(mock_get):
    mock_get.return_value = MagicMock(
        status_code=200,
        json=lambda: {"id": "123", "name": "Test"}
    )
    result = server.tool_name(param1="test")
    assert "id" in result
    assert result["name"] == "Test"
```

#### Why OAuth App Device Flow?

The **Client ID is public** — it can be hardcoded in the skill. The user never pastes a token. The flow:
1. Claude requests a device code → GitHub returns a `user_code` + `verification_uri`
2. Claude shows the user the code and URL — user visits it in their browser and clicks Approve
3. Claude polls until approved → receives a scoped access token
4. Token is used for all API calls and git operations, then discarded

**Required OAuth scopes:** `repo` (create repos + push code) and `write:secrets` (add Actions secrets)

#### 6a — Run the Device Flow and capture the token

```python
#!/usr/bin/env python3
"""oauth_device_flow.py — prints an access token after the user approves in browser."""
import httpx, time, sys

CLIENT_ID = "Iv23litAfxNpFC3nHRhe"  # mcp-from-apidocs OAuth App — public, safe to hardcode
SCOPES = "repo write:secrets"

# Step 1: Request device + user codes
r = httpx.post(
    "https://github.com/login/device/code",
    data={"client_id": CLIENT_ID, "scope": SCOPES},
    headers={"Accept": "application/json"},
    timeout=30,
)
r.raise_for_status()
d = r.json()
device_code = d["device_code"]
interval = d.get("interval", 5)
expires_in = d.get("expires_in", 900)

# Step 2: Show the user what to do (Claude should echo this prominently in chat)
print(f"\nOPEN:  {d['verification_uri']}", flush=True)
print(f"CODE:  {d['user_code']}\n", flush=True)

# Step 3: Poll until approved
deadline = time.time() + expires_in
while time.time() < deadline:
    time.sleep(interval)
    poll = httpx.post(
        "https://github.com/login/oauth/access_token",
        data={
            "client_id": CLIENT_ID,
            "device_code": device_code,
            "grant_type": "urn:ietf:params:oauth:grant-type:device_code",
        },
        headers={"Accept": "application/json"},
        timeout=30,
    ).json()
    if "access_token" in poll:
        print(poll["access_token"], flush=True)
        sys.exit(0)
    err = poll.get("error", "")
    if err == "slow_down":
        interval += 5
    elif err not in ("authorization_pending", "slow_down"):
        print(f"ERROR: {poll}", file=sys.stderr)
        sys.exit(1)

print("ERROR: timed out", file=sys.stderr)
sys.exit(1)
```

#### 6e — Add Docker Hub secrets (optional)

Ask: "Do you also want me to add your Docker Hub credentials as GitHub secrets so CI works straight away?"

If yes, collect `DOCKERHUB_USERNAME` and `DOCKERHUB_TOKEN`, then encrypt and upload with PyNaCl:

```python
import base64, httpx
from nacl import encoding, public

def encrypt_secret(pub_key_b64: str, value: str) -> str:
    pk = public.PublicKey(pub_key_b64.encode(), encoding.Base64Encoder())
    return base64.b64encode(public.SealedBox(pk).encrypt(value.encode())).decode()

TOKEN = "<OAUTH_TOKEN>"
OWNER, REPO = "<GH_USER>", "<api-name>-mcp"

pk = httpx.get(
    f"https://api.github.com/repos/{OWNER}/{REPO}/actions/secrets/public-key",
    headers={"Authorization": f"Bearer {TOKEN}", "Accept": "application/vnd.github+json"},
).json()

for name, val in [("DOCKERHUB_USERNAME", "<dh_user>"), ("DOCKERHUB_TOKEN", "<dh_token>")]:
    r = httpx.put(
        f"https://api.github.com/repos/{OWNER}/{REPO}/actions/secrets/{name}",
        headers={"Authorization": f"Bearer {TOKEN}", "Accept": "application/vnd.github+json"},
        json={"encrypted_value": encrypt_secret(pk["key"], val), "key_id": pk["key_id"]},
    )
    print(f"{name}: {'✅' if r.status_code in (201, 204) else '❌'} ({r.status_code})")
```
